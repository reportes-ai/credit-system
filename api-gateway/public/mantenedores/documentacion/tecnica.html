Documentación Técnica

Arquitectura, tablas, motores e integraciones del Business Suite. Referencia para desarrollo y para el Administrador técnico.

Stack y arquitectura

BackendExpress.js + Node.js — microservicios independientes en /services/*/src/{controllers,routes}, montados por el api-gateway.
Base de datosTiDB Cloud (MySQL-compatible). Pool único compartido: shared/config/database.js. Timezone del pool alineado a Chile (gotcha mysql2: debe coincidir con SET time_zone o las horas se corren 4h).
FrontendHTML/CSS/JS vanilla en api-gateway/public/. PWA instalable (manifest + sw sin caché). App dedicada AF WhatsApp con manifest propio (scope /whatsapp/).
DeployGitHub main → Render auto-deploy (~2-3 min). APP_VERSION en public/js/app-version.js confirma el deploy en el badge de todas las páginas.
AuthJWT (shared/middleware/auth.js). Tokens separados: interno (usuarios), tipo=dealer (portal dealer), tipo=cliente (portal cliente). Permisos por requireFunc('codigo') contra la matriz en BD (Admin bypass, caché 60s). Regla "solo otorgas lo que tienes" (services/usuarios/src/otorgables.js): un no-Admin solo concede/quita funcionalidades que él mismo tiene, no toca el perfil Administrador y no asigna perfiles con permisos superiores a los suyos — enforcement server-side en matriz, masivo, overrides y crear/editar usuario.
MigracionesIIFE (async () => {...})() al inicio de cada controller: CREATE TABLE IF NOT EXISTS + ALTER ADD COLUMN IF NOT EXISTS + seeds INSERT IGNORE. Pendiente: runner central (capataz) que las serialice.

Máximas de diseño (no negociables)

1. Un solo motor por cálculo: cada magnitud se calcula en UNA función compartida que usan todos los consumidores. 2. Una sola fuente de datos: cada dato vive en una tabla autoritativa; el resto la referencia por JOIN/FK. Snapshot deliberado (lo congelado al otorgar) ≠ duplicado. 3. Todo proceso nace documentado: cada feature se documenta al crearla (qué hace y para qué, cómo paso a paso con su pantalla, quién lo hace, prerequisitos, qué procesos afecta y cuáles dependen de él, y su flujo) en estos mismos documentos — un proceso sin documentar está incompleto. 4. Anti-hardcode: módulos y cards salen de las tablas modulos/funcionalidades, nunca de listas JS.

rh_firmasFirma Electrónica Simple (Ley 19.799) de CONTRATO/FINIQUITO: (entidad, entidad_id, rol TRABAJADOR/EMPLEADOR UNIQUE, id_usuario, nombre, cargo, ip, hash_doc SHA-256 del snapshot canónico). Flujo: enviar (RRHH; congela → estado EN_FIRMA, vincula id_usuario por RUT si falta) → firmar TRABAJADOR (solo el titular, desde Mi Ficha) → firmar EMPLEADOR (rh_aprobar) → FIRMADO. Verificación: recalcular hash vs firmado = íntegro/alterado. API /api/rrhh/firmas. También registra las VACACIONES: firma TRABAJADOR al solicitar y EMPLEADOR al resolver. Marco legal: Ley 19.799 arts. 3 y 5 (equivalencia funcional y admisibilidad probatoria de la FES); la evidencia técnica que sustenta cada firma es {identidad: sesión JWT, nombre, cargo, timestamp BD, IP del request, hash SHA-256 del snapshot canónico}; la verificación de integridad recalcula el hash y compara (íntegro/alterado). El finiquito requiere además ratificación art. 177 CT (ministro de fe o finiquito electrónico DT, Ley 21.327); los descuentos de solicitudes se amparan en art. 58 CT (acuerdo escrito = doble firma FES).
